**В проекте реализованы все три уровня, все компоненты работают**

- shared_ui
- shared_feature
- shared_context
- shared_notification (shared_ui, shared_feature)
- auth_mf (shared_feature)
- place_mf (shared_context, shared_feature, shared_ui)
- profile_mf (place_mf, shared_context, shared_feature, shared_ui)
- app_host (auth_mf, profile_mf, place_mf, shared_context, shared_feature, shared_notification)

--------

**Для создания микрофронтендов будет использован webpack module federation**

- Стабильный продукт, имеет большое community, есть репозиторий от создателей с примерами использования, который регулярно обновляют с использованием актуальных подходов и библиотек
- Есть опыт работы с `webpack`, что позволило бы помочь команде в настройке и поддержке
- В проекте используется только react - webpack дает удобную настройку для переиспользования модулей между микрофронтами
  (`shared: [react, react-dom....]`)

- Для работы с UI будет использован `React`, роутинг - `React Router` 
- Для повышения надежности продукта используется `typescript`

**Стратегия проектирования**

На текущий момент понятны бизнес функции каждого домена
Есть возможность выделить независимые блоки
Используется стратегия вертикальной нарезки

В проекте есть несколько основных бизнес компонентов, которые можно сделать независимыми
- Авторизация
- Профиль пользователя 
- Фотографии мест 

Так каждая команда сможет независимо улушать и публиковать свой модуль\
Глубже погружаться в бизнес требования каждой из этих систем

Кроме того необходимо выделить несколько техническо-вспомогательных модулей, которые будут переиспользованы между бизнес компонентами\
Их изоляция позволит например менять подход работы с состоянием приложения.\
Оперативно менять дизайн во всех частях 

```
- shared context        // Доступ к глобальным состояниям приложения (Auth, CurrentUser)
- shared_feature        // Модуль имеет публичный интерфейс и приватную реализацию (Работа с состоянием, публикация событий)
- shared_notification   // Единая нотифкация во всех приложениях
- shared_ui             // UI-kit приложения 
``` 

**В качестве интеграции подходит метод Run time**

Скорее всего, самым популярным сервисом будет работа с фотографиями, так как это основной бизнес-функционал приложения.\
Редактирование профиля будет использоваться явно реже.\
Подход `run time` позволит гибко масштабировать проект и выделять больше ресурсов только на работу с фотографиями

В дальнейшем модуль авторизации может быть изменен - будут введены новые меры безопасности, использование сторонних сервисов и тд
`Run time` позволит независимо обновить этот модуль и не задеть работу команд по работе с фотографиями и редактированием профиля

**Структура проекта**

Каждый микрофнтенд имеет следующую структуру

```
/src
    /components            // React копоненты приложения
        Component.tsx       
    /images                // Ресурсы приложения
        img.jpg
        img.svg
    /models                // Типы модуля - контракт другим командам для безопасного взаимодейтсвия с модулем
        entity.ts
    /styles                // Стили компонентов. Используется БЭМ архитектура на чистом CSS 
        component.css
    /utils
        api.ts              // работа с API
        ... и др 
```

**Управление состоянием**

Для поддержания несвязного соединения между компонентами используется подход `Pub/Sub`

Однако большинству компонентов необходимо знать сосотояния авторизации и текущего пользователя\
Для передачи этих данных будет использоваться `React Context`

**Композиция микрофронтендов**

Для простоты будет использована клиентская композиция\
В дальнейшем следует перейти на гибридный подход, что ускорит работу приложения, улучшит SEO, повысит безопасность

Например, список карточек мест загружается и формирует на сервере, и на клиент уже уходит готовая сетка из фотографий, 
а взаимодействия с карточкой уже происходит на клиенте (просмотр, лайк и тд)

**Текущая структура проекта**

- `app_host`

`App.tsx` - корень проекта, который загружает остальные дочерние компоненты\
определяет роутинг приложения и связывает модули друг с другом\
запускает проверку авторизации пользователя и направляет на страницы Регистрации/Авторизации или на Основную страницу приложения

также содержит кмпоненты `Header` & `Footer`

`Header` - подписывается на состояние авторизации `auth_mf`

- `auth_mf`

авторизация/регистрация пользователя\
проверка валидности пользователя и получение информации о нем (в будущем получение ролей, прав и тд)

так как реализация авторизации может быть разной,\
следует выделить интерфейс по работе с API и реализовать конкретную имплементацию с `nomoreparties`

события авторизации/регистрации публикуются в общую шину\
`app_host` подписывается на эти изменения, что обеспечивает независимость основной логики приложения от реализации авторизации

для нотификации пользователя о результате авторизации/регистрации используется модуль `shared_notification`
связь также реализована через `Pub/Sub` - публикуется событии об открытии нотификации

- `place_mf`

бизнес модуль по работе с фотографиями\
Модуль автономен и реализует всю необходимую логику по работе с фотографими (CRUD операции, лайк фотографии)

Модуль имеет зависимость `shared_ui` для отображения `Popup`. Логика реализована на стороне `place_mf`, визуальная часть в `shared_ui`\
Это позволит иметь единый дизайн во всем приложении и оперативно менять дизайн систему

Также функционал добавления мест выносится в отедельный компонент и будет импортирован в модуле `profile_mf`
`profile_mf` - определяет дизайн кнопки, а `place_mf` содержит реализацию добавления Popup UI + API

- `profile_mf`

бизнес модуль по работе с пользователем\
Модуль автономен и реализует всю необходимую логику по работе с пользователем (Редактирование, смена аватара)

Модуль имеет зависимость `shared_ui` для отображения `Popup`.

`profile_mf` интегрирует в себя модуль `place_mf` для оформления компонента добавления места

***Далее идут shared модули**

- `shared_context`

Доступ к глобальным состояниям приложения (`Auth`, `CurrentUser`) с помощью `React Context`

- `shared_feature`

Предоставления контракта остальным частям приложения для работы с публикацией событий `Pub/Sub` 
и хранению данных на клиенте `localStorage`

Внешне компоненты обращаются к абстрактному API, а внутри уже реализация, которую в дальнейшем можно можно будет поменять, не трогая остальные модули. Скорее всего в реальном приложении для этих фич будут использованы библиотеки и следует эту логику отделить 

- `shared_notification`

Единая нотифкация во всех приложениях

Например для отображения состояния вызова API и др событий

- `shared_ui`

UI-kit приложения 

На текущий момент вынесен только Popup. В дальнейшем следует вынести и др ui компоненты: кнопки, формы, иконки, шрифты и др стандартные элементы дизайн системы. Это позволят хранить консистентность оформления всего приложения и упростить вертску компонент другим командам
